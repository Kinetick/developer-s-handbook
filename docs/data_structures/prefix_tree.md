# Префиксное дерево

[🠔 Оглавление](../index.md#навигация-по-разделам)

**Префиксное дерево (Trie)** — это топологическая, иерархическая структура данных, для которой характерны следующие особенности:

1. Используется для хранения ассоциативных массивов с ключами в виде строк или последовательностей символов.
2. В отличие от [бинарного дерева](./binary_tree.md#бинарное-дерево), узлы префиксного дерева могут иметь неограниченное множество потомков.
3. Все дочерние узлы на одном уровне должны быть **строго уникальными** (ребра между уровнями помечаются различными символами).

> В отличие от бинарного дерева, полное значение ключа не хранится целиком в отдельном узле префиксного дерева. Оно формируется путём последовательного обхода узлов от корня до целевого узла, собирая символы с каждого ребра на своём пути.

Некоторые узлы в дереве помечаются булевым флагом и называются **выделенными**. Например, при обходе по префиксному дереву, хранящему набор слов, попадание в выделенный узел означает, что путь от корня до этого узла образует полное слово. При дальнейшем спуске к дочерним узлам можно получить все слова, начинающиеся с данного префикса.

## Сжатые префиксные деревья (Radix / Patricia)

Префиксные деревья имеют один существенный недостаток — если на каждом уровне гарантирована уникальность ребер, то при наличии узлов с единственным потомком возникает избыточное потребление памяти. Иными словами, можно получить ситуацию, изображённую ниже:

```text
        root
          |
          к
          |
          о
          |
          р
        /   \
     ень      а
```

Это критично, если исходный набор данных содержит большое количество элементов — такие деревья «раздувают» потребление памяти. Для борьбы с этим используют **сжатие**.

### Radix-дерево

**Radix-дерево** (радикс-дерево) — это обобщённое название для сжатых префиксных деревьев, в которых цепочки узлов с единственным потомком объединяются в один узел, хранящий строку символов:

```text
        root
          |
        кор
        /   \
     ень      а
```

### Patricia-дерево

**Patricia trie** (Practical Algorithm To Retrieve Information Coded In Alphanumeric) — это конкретный алгоритм сжатия Radix-дерева, который использует **bitmap** для выбора пути и позволяет хранить ключи целиком, а не посимвольно. Каждый узел Patricia содержит:

- Bitmap, указывающий, какие позиции в ключе значимы для ветвления.
- Ссылки на дочерние узлы, соответствующие установленным битам.

> **Кратко**: Radix tree — это семейство сжатых префиксных деревьев. Patricia trie — один из алгоритмов реализации Radix дерева с оптимизацией выбора пути через bitmap.

## Сложность операций

Ниже приведена оценка временной и пространственной сложности основных операций над префиксным деревом.
*Формат: `Время | Пространство`*

> - **Длина ключа** — `k`.
> - **Размер алфавита (максимальное число потомков узла)** — `σ`.

| Операция | Описание | Временная сложность | Пространственная сложность | Примечание |
| --- | --- | --- | --- | --- |
| **Добавление ключа** | Вставка нового слова в дерево | `O(k)` | `O(k)` | В худшем случае создаётся новый узел для каждого символа ключа. |
| **Удаление ключа** | Удаление существующего слова из дерева | `O(k)` | `O(1)` / `O(k)*` | Простой Trie: обнуление ссылок `O(1)`. Сжатое дерево: каскадное удаление/разделение узлов может потребовать временной памяти `O(k)`. |
| **Поиск ключа** | Проверка наличия полного совпадения | `O(k)` | `O(1)` | Последовательный обход символов ключа от корня. |
| **Проверка префикса** | Поиск всех слов, начинающихся с заданного префикса длины `m` | `O(m + r)` | `O(r)` | Где `r` — количество найденных совпадений. |
| **Поиск наибольшего префикса** | Нахождение самого длинного префикса ключа, присутствующего в дереве | `O(k)` | `O(1)` | Обход до первого узла, где путь расходится с деревом. |

## Реализация в CPython

В **Python** нет встроенной структуры для представления префиксных деревьев. Традиционно их реализуют самостоятельно, например:

<details>

```python
class TrieNode:
    def __init__(self):
        self.children: dict[str, TrieNode] = {}
        self.is_end_of_word = False


class Trie:
    def __init__(self):
        self.root = TrieNode()

    # Интерфейсы поиска и т.д.
```

</details>

### Нюансы реализации

1. **Выбор контейнера для потомков**: Использование `dict` даёт гибкость (поддержка любого алфавита), но добавляет накладные расходы на хэширование. Для ограниченного алфавита (например, ASCII) эффективнее использовать массив фиксированного размера — доступ по индексу работает за `O(1)` без вычисления хэша.

2. **Память под пустые узлы**: В классической реализации каждый символ ключа создаёт отдельный узел. Для алфавита Unicode это может привести к значительному потреблению памяти, так как размер словаря потомков в каждом узле потенциально огромен. Решение — использовать сжатие (Radix/Patricia) или `dict` вместо массива.

3. **Сравнение символов**: В отличие от хэш-таблиц, префиксное дерево требует посимвольного сравнения ключей. Для Unicode-строк важно учитывать нормализацию и регистрозависимость — без предварительной обработки `А` и `а` будут считаться разными путями в дереве.

4. **Параллелизм**: Чтение из префиксного дерева безопасно для многопоточности, так как не изменяет структуру. Запись требует блокировки на уровне узла или использования атомарных операций (например, через `threading.Lock` или `multiprocessing`).

## Префиксные деревья vs BST

Обе структуры подходят для хранения упорядоченных данных, но имеют принципиальные различия:

| Критерий | Префиксное дерево | Бинарное дерево поиска (BST) |
| --- | --- | --- |
| **Сложность поиска** | `O(k)` — зависит от длины ключа | `O(log n)` — зависит от количества элементов |
| **Тип данных** | Только строки/последовательности | Любые сравнимые типы |
| **Префиксные запросы** | Поддерживаются нативно | Требуют обхода и фильтрации |
| **Память** | Высокое потребление (много узлов) | Компактнее при сбалансированном дереве |
| **Удаление** | Сложная логика (каскадное удаление) | Проще реализуется |

> **Вывод**: Если ключи — строки и важны префиксные запросы, Trie выигрывает. Если данные разнотипные или требуется строгий порядок — BST предпочтительнее.

## Практическое применение

Префиксные деревья активно используются в следующих задачах:

- **Автодополнение** — поиск подсказок по введённому префиксу (поисковые системы, IDE).
- **Проверка орфографии** — быстрый поиск слова в словаре.
- **Маршрутизация IP-адресов** — longest prefix matching для определения маршрута.
- **URL-маршрутизация** — сопоставление запросов с шаблонами путей (веб-фреймворки).
- **Поиск по словарю** — генерация анаграмм, слов из заданных букв.
- **Фильтрация контента** — быстрая проверка на наличие запрещённых префиксов/слов.

## Префиксные деревья vs Хэш-таблицы

При всей своей простоте префиксные деревья не всегда являются лучшим выбором при поиске или проверке вхождения. Основная причина кроется в оптимизации потребления памяти на хранение всей структуры.

**Префиксное дерево** будет хорошим выбором, если:

- Требуется найти наибольший (самый длинный) префикс ключа.
- Требуется найти наименьший префикс.
- Необходимо реализовать автозаполнение (автодополнение).
- Необходимо реализовать механизм проверки правописания.

**Хэш-таблицы** будут более оптимальным выбором, если:

- Требуется хранить сильно разреженные ассоциативные массивы (случайные строки).
- Необходим поиск точных совпадений.
