# Хэш-таблица

**Хэш-таблица** — это структура данных, для которой характерны следующие особенности:

1. Объекты хранятся в формате *key:value*.
2. Мгновенная скорость получения объекта по ключу.
3. Мгновенная скорость вставки объекта по ключу.
4. Всё множество ключей в хэш-таблице **строго** уникально.

> В общем виде хэш-таблица состоит из двух компонентов: массива фиксированного размера для хранения данных и хэш-функции. Хэш-функция преобразует произвольный ключ (строку, число) в индекс массива:
>
> ```txt
> index = hash(key) % array_size
> ```
>
> Поскольку пространство возможных ключей бесконечно, а размер массива конечен, неизбежно возникают **коллизии** (ситуации, когда два разных ключа выдают один и тот же индекс).

---

## Реализация в CPython

В более современных (3.6+) версиях **Python** хэш-таблица состоит из трех важнейших элементов: хэш-функции, разреженного массива индексации и плотно упакованного массива значений. Первые два элемента сильно пересекаются со [множествами](./set.md#множество-cpython).

### Структура данных

- В массиве индексации хэш-таблицы хранятся **индексы** на ячейку со структурой в плотно упакованном массиве значений.
- Массив индексации более легковесен по сравнению с аналогичным в множествах.
- Массив значений не содержит «дыр»: структуры объектов записываются туда последовательно по мере добавления в словарь.

### Сохранение порядка

Такой архитектурный подход решил одну важную проблему множеств — **сохранение порядка вставки элементов**. Поскольку значения добавляются в плотный массив последовательно, их порядок строго соответствует порядку добавления в словарь.

---

## Сложность операций

Ниже приведена оценка временной и пространственной сложности основных операций.
*Формат: `Время | Пространство`*

> - **Длина основного словаря** — `n`.
> - **Длина вспомогательного словаря** — `k`.

| Операция | Метод / Синтаксис | Временная сложность | Пространственная сложность | Примечание |
| --- | --- | --- | --- | --- |
| **Добавление объекта** | `dict[key] = value` | `O(1)` | `O(1)` | Вычисление хэш-значения `O(1)`, переход и запись по индексу в массив индексации `O(1)`, запись в конец массива значений `O(1)`. |
| **Удаление объекта** | `del dict[key]` | `O(1)` | `O(1)` | Вычисление хэш-значения `O(1)`, запись dummy-указателя в массив индексации `O(1)`, зануление ячейки в массиве значений `O(1)`. |
| **Получение объекта** | `dict[key]` | `O(1)` | `O(1)` | Переход по индексам в двух массивах. |
| **Объединение** | `dict1 \| dict2` | `O(n + k)` | `O(n + k)` | Создается новый объект. Копируются массивы первого словаря `O(n)`, затем добавляются элементы второго `O(k)`. |
| **Объединение на месте** | `dict1.update(dict2)` | `O(k)` | `O(k)` | Новый объект не создается. В худшем случае (нет пересечений ключей) сложность `O(k)`. При полном пересечении ключей пространственная сложность падает до `O(1)`. |
| **Копирование словаря** | `dict.copy()` | `O(n)` | `O(n)` | Создается новый объект, физически обходятся и копируются массивы индексации и значений. |
| **Итерация** | `for k in dict` | `O(n)` | `O(1)` | Обход внутреннего массива значений. Массив может быть больше количества хранимых структур из-за удаленных элементов (решается алгоритмом сжатия). |
| **Проверка вхождения** | `key in dict` | `O(1)` | `O(1)` | Аналогично получению значения. |
| **Получение длины** | `len(dict)` | `O(1)` | `O(1)` | Длина всегда актуальна и хранится в метаструктуре. |

## Словарь vs Множество

Если стоит задача только проверить вхождение значения и соблюсти уникальность, то **множество** будет предпочтительнее. Множество поддерживает ряд операций из теории множеств, которых нет у словарей. Однако, если нужно сохранить уникальность **и** порядок входного списка/кортежа, то **словарь** станет идеальным выбором.

Ниже приведены два примера кода, оба позволяют отфильтровать дубликаты и сохранить исходный порядок:

```python
# Множества
input_list = [...]
result = list(sorted(set(input_list), key=input_list.index))

# Словарь
input_list = [...]
result = list(dict.fromkeys(input_list))
```

### Анализ производительности

Ситуация не кажется критичной при небольшом количестве значений, но если входной список имеет $10^7$ элементов, первый способ провалится.

1. **Множества**: Функция `input_list.index` — это **линейный поиск** со сложностью `O(n)`. В итоге общая сложность первого способа уходит в **`O(n^2)`**.
2. **Словарь**: Использует свойства хэш-таблицы, работающие за `O(1)`. Поскольку `None` — это синглтон, в куче памяти висит лишь одно значение, а в худшем случае $10^7$ ссылок на него.

По итогу первый способ может «повесить» процесс, тогда как второй сможет обработать данные до тех пор, пока не закончится память под внутренние структуры (массивы индексации и значений).
