# Массив

Массив это структура данных, для которой характерны следующие особенности:

1. Непрерывная область в памяти, в отличие от связанного списка
2. Элементы массива **всегда** одного типа
3. Мгновенное получение элемента по его индексу
4. Мгновенная замена значения элемента по индексу

Мгновенные операции получения/изменения элемента по индексу вытекают из п.1 и п.2. Для Си подобного языка в переменной массива хранится указатель на самый первый элемент с индексом 0. Если область памяти непрерывна и тип всех элементов одинаков, значит через адресную арифметику перейти к нужному участку памяти это дешевая операция.
Например, есть массив из 10 элементов типа int_8t (1 байт). Для получения пятого элемента (индекс 4) нам нужно вычислить его адрес в памяти по формуле:

```txt
адрес = указатель на начало массива + 4 * размер(int_8t)
```

За высокую скорость доступа к элементам приходится платить - любая операция по удалению элемента, кроме как из конца массива требует дефрагментирования памяти. Если удалить элемент из середины массива то потребуется сдвинуть **все** элементы после удаляемого влево на одну позицию с последующим усечением конца массива.

## Список CPython

В **python** список это массив указателей на значение в памяти. Это важно понимать, массив хранит **указатели** (адреса в памяти), а не сами значения. Именно поэтому список способен хранить разнородные структуры внутри себя. Физически значения располагаются в куче памяти, но из-за того что известен адрес можно мгновенно получить его.

Раз список это по сути массив значит он имеет все преимущества последнего и разделяет его недостатки. Чтобы скомпенсировать проблемы массивов из Си, разработчики python используют оптимизацию - **избыточное выделение памяти**. Оптимизация нужна для амортизации выделения памяти при добавлении элемента в конец списка. Избыточное выделение рассчитывается динамически по формуле CPython. На малых объемах оверхед составляет около 30–60%, но с ростом списка процент избытка падает до ~12.5%, чтобы эффективно балансировать между скоростью добавления и экономией RAM.

### Сложность операций

При описании сложности используется формат: *\[временная, пространственная\]*.
Реальный размер массива - *size*.
Выделенная память в непрерывном блоке под указатели - *allocated*.

Операции:

- Добавление в конец (`append`) **\[O(1), O(1)\]** - в худшем случае, если *size >= allocated* временная сложность вырастет до **O(n)** за счет тяжелой функции *ralloc*.
- Удаление с конца (`pop(-1)`) **\[O(1), O(1)\]** - освободившийся слот памяти не возвращается в систему, а резервируется в качестве амортизатора.
- Добавление в произвольное место (`insert`) **\[O(n), O(1)\]** - самый худший вариант это вставка в голову списка. Здесь придется сдвинуть все элементы в позиции \[1:\] на один индекс вправо.
- Удаление из произвольного места (`pop`) **\[O(n), O(1)\]** - более общий случай удаления из головы списка.
- Расширение (`extend`) **\[O(k), O(k)\]** - если *size < allocated* пространственная сложность может упасть до **O(1)**. Если *size >= allocated* то временная сложность может возрасти до **O(n+k)**
- Итерация (`for x in list`) **\[O(n), O(1)\]**.
- Получение среза (`[x:y]`) **\[O(k), O(k)\]** - перейти на первый элемент среза **O(1)**, затем потребуется итерироваться k раз **O(k)** и выделить память под новый список **O(k)**.
- Вставка среза в произвольное место (`[x:y] = [x1:y1]`) **\[O(n+k), O(n+k)\]** - перейти в место вставки **O(1)**, копирование новых элементов **O(k)**, последующий сдвиг **O(n)** в худшем случае. Амортизация может сгладить пространственную сложность при маленьком срезе до **O(1)**.
- Удаление среза (`del [0:x]`) **\[O(n), O(1)\]** - худший сценарий это удаление среза с головы списка. Придется двигать все элементы справа на лево.
- Умножение (`l * k`) **\[O(nk), O(nk)\]** - является частным случаем расширения списка.
- Сортировка (`sort`) **\[O(nlogn), O(1)\]** - в лучшем случае за счет стабильности сортировки *Timsort* отработает за *O(n)* временной сложности и **O(1)** пространственной. В худшем случае потребуется дополнительная память для хранения подсписков для слияния и пространственная сложность возрастет до **O(n)**.
- Получение по индексу (`[x]`) **\[O(1), O(1)\]** - адресная арифметика.
- Проверка на вхождение (`x in list`) **\[O(n), O(1)\]** - физически придется пройтись по всем элементам.
- Копирование (`copy`) **\[O(n), O(n)\]** - физически придется пройтись по всем элементам и скопировать их. Важно учесть, что это поверхностное копирование - сами данные в куче не копируются.
- Поиск минимума/максимума (`min/max`) **\[O(n), O(1)\]** - физически придется пройтись по всем элементам.
- Получение длинны (`len`) **\[O(1), O(1)\]** - длинна списка хранится в отдельной структуре метаданных и всегда актуальна, поэтому не считается каждый раз.

### Кортеж vs Список

Физически кортеж — это тоже массив указателей на объекты в куче, разделяющий со списками **O(1)** на чтение по индексу. Однако за счет неизменяемости (immutability) кортежи оптимизированы на уровне ядра CPython:

- **Отсутствие избыточной памяти**: Лимит зарезервированных слотов всегда равен длине.
- **Inline-аллокация**: Метаданные и указатели на элементы лежат в одном непрерывном блоке Си-структуры.
- **Free Lists**: Уничтоженные кортежи (до 20 элементов) кэшируются в памяти для мгновенного переиспользования при создании новых.
- **Мгновенное копирование**: Python не будет ничего копировать, он просто вернет ссылку на тот же самый старый кортеж и увеличит его счетчик ссылок. Операция займет мгновенное **O(1)** по времени и **O(1)** по памяти.
